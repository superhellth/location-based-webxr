Development Workflow: Plan First, Then Prototype
A core idea in software engineering is to build software out of small, independent components with clear interfaces between them. Each component is developed, tested, and maintained on its own, and only integrated with the others once it is complete and proven to do what it should. This keeps complexity manageable, lets your team work on different components in parallel, and — crucially — means any single component can later be replaced with an improved version without touching the rest of the app, as long as its interface stays the same.
The Plan First, Then Prototype workflow fits naturally into this approach: it lets you nail down and validate one component at a time in isolation. It also makes AI-generated code far less risky — if an LLM produces a component that works but is a bit too verbose or just not maintainable over time, that stays an isolated, replaceable problem of that one component (you can redo or swap out just that piece of the entire system) rather than something that eventually forces a rewrite of the whole application.
Before building any new component, follow this workflow (it builds on the plan-first approach you already practiced in Phase 2). Don't jump straight into writing the production code in your repo.
Write a detailed plan first. For each component, create a plan markdown file in the repo (e.g. 2026-12-24-PLAN.md or 2026-12-24-<component>-plan.md) that captures, before any production code:
what the use case is about and the problem it solves,
what you want to achieve — the concrete goals, requirements, and how you will know it works (success criteria),
the intended approach / architecture and the key open questions.
It is fine — and encouraged — to include example code snippets in the plan to make an idea concrete.
Iterate on the plan with an LLM. Feed the plan to an LLM and ask it to act as a critical reviewer: poke holes, surface edge cases, and propose alternatives (the same "make the AI challenge you" skill from Phase 2). Do a few rounds and refine the plan until no major problems are left. Commit each meaningful revision so the plan's evolution is visible in git.
Generate several throwaway prototypes. Once the plan is solid, hand the same plan to multiple different LLMs and have each one build a small, self-contained, testable prototype — just enough to run and try, not the final system. You can do this either in VS code, in the Gemini or OpenAI web uis or any other tool you prefer where web examples can efficiently be prototyped and tested.
Try them and pick the best. Run the prototypes and compare them on both axes: the UX (how it looks and feels to use) and the code (is it clean, understandable, and reasonable to build on?). From this review you will typically be able to pick one you prefer most as a starting point.
Build the real thing. Treat the winning prototype as reference/example code — do not just drop it into the repo as-is. Implement a proper, reusable, well-tested version in your actual repository, following the architecture and separation-of-concerns expectations you already know.
Test it at two levels. Cover the component's pure functions and logic with unit tests — they are fast and pin down the logic, math and edge cases. On top of that, add end-to-end tests ("e2e" tests) that replay the realistic outdoor recordings you captured in Task 1, so you can verify the whole component behaves correctly in a real outdoor context — deterministically, on a desktop, without manual input by you every time you do a small change. Unit tests get you far, but the replay-driven e2e tests are what prove the system actually works in the field and that everything is plugged in together correctly.

2.1 The product in one paragraph (the destination)
An author (e.g. castle staff) walks a route once and uses an authoring mode to drop waypoints, attach assets (2D sprites, GLTF/GLB 3D models, audio files) to those waypoints, and record a breadcrumb trail between them by physically walking it. The app exports everything into a single tour.zip (a tour.json description plus the asset files). The author uploads that zip to an everyday cloud-storage service they already use (Google Drive, Dropbox, OneDrive, Box) and generates a QR code whose URL opens the app in viewing mode pointed at the zip's download URL (https://app.com/?tour=https://…/tour.zip). A visitor scans the QR code, the viewing mode opens the tour zip straight from its URL — reading the small tour.json and then each asset out of it on demand via byte-range requests, without downloading the whole archive — and then guides them along an AR breadcrumb trail of glowing orbs. As the visitor approaches a waypoint, an object (e.g. a knight) appears in the AR view: its assets are quietly pre-loaded while the visitor is still far away, it becomes visible once they are close, and — when the visitor taps it — it tells a historical story (audio + a floating text transcript for noisy outdoor use). A toggleable 2D map shows real-time position and points of interest. There is no backend database — the zip contains the full state and assets of the tour.
2.2 The architecture that makes this splittable (agree on and document this first)
Everything in this app flows through one shared contract, and getting this contract right before you split up the work is what lets the two of you build different components that can afterwards be combined correctly:
The tour.json schema — the on-disk/on-wire format. Waypoints (lat/lon, the proximity radii, ordering), points of interest and which asset files they reference, and the breadcrumb trail points. This is the data the authoring side writes and the viewing side reads. Note that lat/lon is only how a location is persisted; at runtime location-based-webxr anchors each one to an ordinary world-space position in meters, and everything downstream (proximity, the scene) works in world space, not geo coordinates — see §2.5.1.
The Redux store — the single source of truth at runtime for all three worlds: the 2D DOM UI, the Three.js/WebXR scene, and the live GPS/device state. The same lesson as Phase 1: business logic and state are separated from the view, and both the 2D UI and the 3D scene subscribe to the store. Assets at runtime are referenced by id (the name/key from tour.json), never by file path. The store does not hold the asset bytes — it goes through a small asset-provider interface (e.g. getAssetUrl(id) → Promise<url> and release(id)) that hands out an in-memory Blob URL when an asset is needed and frees it again afterwards. Hiding the bytes behind that interface is what lets you swap the loading policy later (see §2.5.4) without touching the scene or the store.
Write this contract down in your first plan file (TypeScript interfaces for tour.json and for the relevant slices of Redux state) and review it with us early. Once it is agreed, each component below talks only to this contract, so you can build and test them independently.

2.5 Recommended approaches & the hard parts (these double as search terms)
Work in world space, not geo coordinates. location-based-webxr already fuses GPS + IMU and anchors content into the ordinary Three.js scene as world-space positions in meters — so do not write haversine/equirectangular distance math yourself, and do not pass lat/lon into your proximity or scene logic. The distance between the user and an object is just userPos.distanceTo(obj.position) on two THREE.Vector3s, exactly as in any non-AR Three.js scene. The only places geographic coordinates legitimately appear are the persisted tour.json and the single anchoring step the framework performs when a tour loads. Understand how the library exposes the user's pose and how you register an anchor before building on top. (Search: "location-based-webxr anchor", "three.js Vector3 distanceTo".)
Forgiving wayfinding. Real GPS is 1–2 m off and the compass drifts. Do not fight this with precise arrows to an exact point — the spec's breadcrumb orbs deliberately guide to the general vicinity of the next waypoint (its ≈10 m active radius) and let the visitor's own eyes do the final approach. Build in hysteresis everywhere a threshold is crossed (see component 4).
Avoiding jank ("the main thread froze"). Parsing/instantiating a GLTF model is expensive; doing it the instant a knight should appear causes a visible frame-rate stutter mid-experience. That is the entire reason for the 25 m PREFETCH zone: ask the asset-provider for the model, parse it, and instantiate it invisibly while the user is still far away, so the only thing the 10 m ACTIVE transition has to do is flip visible = true. (Search: "GLTFLoader parse blob", "Three.js avoid jank model loading".)
Asset memory: read each asset on demand by byte range, and dispose what the GPU holds. Because the zip is stored uncompressed (component 5) there is no decompression step and no need to hold the whole archive in memory. Read the zip's central directory once, then pull each asset's bytes on demand — fetching just that entry's byte range straight from the hosted URL with HTTP Range requests. Use a range-capable zip reader (e.g. @zip.js/zip.js's HttpRangeReader) against a host that supports Range + CORS (§2.5.6); vanilla JSZip is not suitable here because it pulls the whole buffer into memory. That leaves two tiers of memory to manage, both freed on walk-away (a waypoint dropping back to IDLE):
The parsed Three.js GPU resources (geometries + textures in VRAM) — usually the largest and the first thing that gets a mobile tab killed, and the part uncompressed storage does nothing for, so this is the real work. Dispose them (geometry.dispose(), material.dispose(), texture.dispose()).
The asset's Blob / Blob URL — a Blob URL is only a handle, the underlying Blob stays resident until you revokeObjectURL it.
The PREFETCH→ACTIVE window exists to hide the fetch + parse cost when a knight is approached. Avoid thrash at the boundary: a visitor pacing across the 25 m line would otherwise re-fetch + re-parse repeatedly; the state-machine hysteresis already softens this, and on top of it a small LRU cache of the last few parsed models keeps recently-seen knights from being rebuilt constantly. All of this lives behind the §2.2 asset-provider interface, so the scene and store never change. Warm a local cache in the background, then switch byte-source from remote to local. Range requests and a full download are complementary, not either/or — combine them. Have the zip reader sit behind a tiny byte-source interface (read(offset, length) → bytes) so its source is swappable. Start with the remote source: range-fetch the central directory and then whatever the visitor is near, so the first knights are available instantly with no upfront wait. At the same time, kick off a low-priority background download of the entire zip into a persistent cache (Cache API or IndexedDB). When that finishes, transparently swap the byte-source from "remote Range fetch" to "read from the local cached copy" — mid-session, with no change visible to the scene or store. From then on every asset read is local: no network latency, no per-asset requests, and the tour keeps working if connectivity drops. Keep the background download de-prioritised so it does not starve the on-demand range fetches the visitor is actively waiting on, and make the switch idempotent (in-flight remote reads can complete from remote; only new reads go local). (Search: "Cache API store response offline", "IndexedDB store ArrayBuffer", "fetch priority low background download".)
The audio-unlock gesture. Browsers will not let you start audio without a user gesture. The Start button in the onboarding gate is that gesture — create/resume the AudioContext in that click handler. If you try to play a knight's story later without having done this, it will silently fail. (Search: "Web Audio autoplay policy unlock AudioContext user gesture".)
Hosting on everyday cloud storage. The whole point of the serverless model is that an author can publish a tour using a service they already have — Google Drive, Dropbox, OneDrive, or Box — with no servers of their own. Research shows all four support partial (Range) downloads, which is what the byte-range reading in §2.5.4 depends on, but each has its own quirks, and the URL that supports ranges is the service's download/content URL, not the human "share" page:
Google Drive: range works on the API media URL (files.get?alt=media), not on a plain share link, and not on native Google Docs/Sheets (only on real uploaded binary files like a .zip).
OneDrive / Box: the content endpoint typically answers with a 302 redirect to a temporary download URL — your client must follow the redirect and re-apply the Range header to the final URL.
Dropbox: range works on the content-download endpoint (content.dropboxapi.com), but not on Dropbox's own "download folder as zip" feature — so the author must upload one ready-made .zip file, not a folder.
The single hardest requirement for a browser-only app is CORS: whichever download URL you settle on must also return Access-Control-Allow-Origin for your app's origin and expose Accept-Ranges/Content-Range, or the browser will block the cross-origin range read regardless of whether the server itself supports ranges. This is the thing to verify first, before building on top of a given provider.
Practical checks: a HEAD (or a tiny Range: bytes=0-0) request that comes back with Accept-Ranges: bytes / 206 Partial Content and the right CORS headers tells you a URL is usable; a 200 with the whole body, or a CORS error, tells you it is not. Fetch the file's size first so you never ask for a range past the end (416 Range Not Satisfiable). Pick one provider, get it working end-to-end, and document the exact "how to upload + get the right link" recipe so an author can actually publish a tour.
(If a tour is published behind a link that turns out to serve the whole file but not ranges, the app should still work — fall back to the "download the full zip, then read locally" path from §2.5.4; you just lose the instant-first-asset benefit.) (Search: "Google Drive files.get alt=media range header", "OneDrive Graph downloadUrl range request", "Dropbox content api range header", "CORS Accept-Ranges Content-Range expose-headers".)
QR contents. The QR encodes the app's URL with the zip URL as a query parameter — keep the zip URL short (long URLs make denser, harder-to-scan codes). (Search: "QR code generator JS", "url shortener for QR density".)
